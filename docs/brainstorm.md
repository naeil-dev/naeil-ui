# Brainstorm: Design System

**Date:** 2026-02-27

## Idea

> "나만의 아이덴티티가 있는 디자인 프레임워크/기반 — 다양한 UI 및 프로그램을 만들 때 모든 것의 바탕이 되는 디자인 시스템을 만들고 싶다. 공유도 가능하게."

## Key Questions & Answers

| Question | Answer |
|----------|--------|
| Who is this for? | 본인 + 공유 (npm 배포 or 오픈소스) |
| What problem does it solve? | 프로젝트마다 디자인 처음부터 → 일관된 아이덴티티 + 재사용 |
| Why now? | 다양한 UI/프로그램을 만들기 시작하는 시점 |
| Existing solutions tried? | 없음 (백지 상태) |
| Constraints? | 1인 개발, 디자이너 없음, 무료 운용 원칙 |

## 취향 프로파일 (ENTJ)

**좋아하는 것:** 미니멀, 깔끔, 기계적, 전문성, 다크모드
**싫어하는 것:** 따뜻함, 부드러움, "라이프스타일" 느낌 (Craft ❌)

### 레퍼런스 사이트 (Jay 선택)
- ✅ **Linear** (linear.app) — 극도의 미니멀, 모노크롬
- ✅ **Vercel** (vercel.com) — 흑백, 기계적 정밀함, Geist 타이포
- ✅ **Warp** (warp.dev) — 다크, 개발자 도구, 기계적
- ❌ Craft (craft.do) — 따뜻하고 부드러움

### 추출된 디자인 DNA
| 특성 | 방향 |
|------|------|
| 베이스 톤 | 모노크롬 다크 |
| 악센트 색상 | 단일 악센트, 극소량 |
| 타이포 | 기하학적 산세리프, 볼드 헤딩 |
| 여백 | 콘텐츠 밀도 > 과도한 여백 |

| 장식 | 제로 |
| 느낌 | 도구의 미학 — 정밀함 > 따뜻함 |

## Assumptions

### Validated
- 디자인 시스템은 "철학/원칙 → 토큰 → 컴포넌트" 순서로 구축 (업계 합의)
- W3C DTCG 토큰 표준 v1이 2025.10에 안정화됨 — 따르면 미래 호환성 확보
- React가 2026년에도 가장 큰 생태계 + AI 코드 생성 최적화
- shadcn/ui는 Radix(접근성) + Tailwind(스타일)의 조합이고 코드 소유 모델

### Challenged
- "처음부터 100% 독자적으로 만들어야 한다" → ❌ 솔로 개발자에게 비현실적. shadcn을 출발점으로 커스텀이 효율적
- "컴포넌트 라이브러리까지 한 번에 만들어야 한다" → ❌ 토큰+테마 먼저, 컴포넌트는 실제 프로젝트 하면서 점진적 추가
- "프레임워크 독립적이어야 한다" → 이상적이지만 현실적으로 React에 집중하는 게 효율적

## Options Explored

### 프레임워크
| Option | Pros | Cons | Complexity | Risk |
|--------|------|------|------------|------|
| React + Next.js | 최대 생태계, 레퍼런스 3/3 일치, AI 최적 | 약간의 학습곡선 | Low | Low |
| Svelte | 가벼움, 쉬움 | 컴포넌트 생태계 부족 | Med | Med |
| Vue | 쉬움, 안정 | 레퍼런스 불일치, 생태계 작음 | Low | Med |

### 컴포넌트 접근법
| Option | Pros | Cons | Complexity | Risk |
|--------|------|------|------------|------|
| A: shadcn/ui 커스텀 | 빠름, 접근성, 소유권 | 기본 스타일 교체 필요 | Low | Low |
| B: Radix + 직접 스타일 | 100% 고유 아이덴티티 | 느림, 디자인 감각 의존 | High | Med |
| C: shadcn → 점진적 교체 | 실용적, 바로 사용 가능 | 아이덴티티 확립 느림 | Low | Low |

## Devil's Advocate Findings

1. **디자인 시스템 자체가 목적이 될 위험** — 실제 프로젝트 없이 시스템만 다듬다 끝나는 흔한 패턴. 반드시 실제 프로젝트와 병행
2. **디자이너 없이 "아이덴티티"를 만드는 건 어렵다** — 색상/타이포 조합은 감각 영역. 시행착오 필요. 미니멀+기계적을 원하는 건 다행 (장식 미니멀이 실현 가능)
3. **공유 의도 = 문서화 부담** — 혼자면 문서 불필요, 공유하려면 사용법/가이드/예시 필수

## Decision

**프레임워크:** React + Next.js + TypeScript
**CSS:** Tailwind CSS
**컴포넌트:** Option C — shadcn/ui 시작 → 점진적 커스텀
**토큰 표준:** W3C DTCG 준수
**테마:** 다크 우선 (Dark-first) + 라이트 모드 지원

**Reason:** 가장 큰 생태계 + 레퍼런스 일치 + 즉시 사용 가능 + 점진적 아이덴티티 확립. 1인 개발자가 현실적으로 실행 가능한 최적 조합.

**Deferred:**
- 프레임워크 독립 Web Components 버전
- 전용 서체 제작 (Vercel Geist처럼)
- Figma 디자인 키트
